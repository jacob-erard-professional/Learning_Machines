# Implementation Plan: Voice Detection Form Input

**Branch**: `feat/voice-detection-form-input` | **Date**: 2026-03-21
**Spec**: `specs/feat/voice-detection-form-input/`

## Summary

Two complementary voice features sharing a common `useVoiceCapture` base hook:

1. **Conversational AI Intake (user-side)** — A chatbot modal where requestors speak or type naturally ("We're hosting a school health fair with 200 kids"). Claude extracts multiple fields from each message, has a warm conversational style, and guides the user until all 8 required fields are collected. Ends with an editable review step before pre-filling the `RequestForm`.

2. **Voice Command Interface (admin-side, high impact / low effort)** — A compact bar on the admin queue page. Admin says "Show high priority events this week" → Claude interprets the command → queue filters update instantly. Uses Web Speech API (no model download, streaming, instant response).

**Technical approach**: HuggingFace Transformers.js (Whisper-tiny.en, in-browser WASM) for speech-to-text + Claude API with JSON-mode extraction + in-memory session state on Express backend. Voice is progressive enhancement — chat input always available as fallback.

## Technical Context

**Language/Version**: Node.js 20 LTS (backend), React 18 + Vite 5 (frontend)
**Primary Dependencies**: Anthropic SDK (`@anthropic-ai/sdk`), React Testing Library + Vitest (frontend tests), Jest + Supertest (backend tests)
**Storage**: In-memory `Map` for voice sessions (swap target: `lowdb` for persistence)
**Testing**: Vitest + React Testing Library (frontend), Jest + Supertest (backend)
**Target Platform**: HTTPS web app (required for MediaRecorder + WASM in production; localhost works in dev)
**Project Type**: Web application (React frontend + Express backend)
**Performance Goals**: < 2s round-trip from voice stop → AI reply displayed; Whisper inference ≤ 8s before fallback triggers
**Constraints**: MediaRecorder + WebAssembly required; Claude API key required; voice is progressive enhancement — if WASM/Worker unsupported OR inference exceeds 8s, mic button is hidden and user types in chat input directly; modal header MUST display "Your responses are processed by AI to fill your form" (no action required from user)
**Scale/Scope**: Single session per open modal; sessions cleaned on close/completion
**Correction UX**: Users correct wrong extractions mid-conversation via natural language; when `isComplete: true`, a review step renders all 8 fields as editable inputs; edits update local state only (no further Claude calls); `onComplete` fires with the final merged values
**Session Seeding**: On modal open, any already-filled `RequestForm` fields are passed to `POST /api/voice-intake/start` as `initialFields`; Claude skips those and asks only for missing ones

## Clarifications

### Session 2026-03-21

- Q: What is the fallback strategy when Whisper inference is too slow or WASM unsupported? → A: Voice is progressive enhancement — if WASM/Worker not supported OR inference exceeds 8s, hide the mic button and let the user type in the chat input. The conversation with Claude works identically in both modes.
- Q: Does the healthcare context require a user consent notice before AI processes their conversation? → A: Show a one-line inline notice in the modal header — "Your responses are processed by AI to fill your form" — no user action required.
- Q: Can the user correct a wrongly-extracted field mid-conversation? → A: Both — user can correct by speaking/typing mid-conversation ("actually my zip is 84109") AND when isComplete triggers, a review step shows all 8 fields as editable inputs before the user confirms and pre-fills the form.
- Q: Should existing form values seed the voice session when the modal opens? → A: Yes — pass any already-filled form fields to POST /api/voice-intake/start so Claude pre-seeds those fields and only asks for missing ones.
- Q: Does editing a field in the review step re-engage Claude? → A: No — the review step is a standalone editable form; edits update local state only, then onComplete fires with the final merged values. No further API calls after isComplete.
